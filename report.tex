\section{Introduction}
When machine learning models are used for decisions like loan approvals or recidivism
prediction, they can end up treating people differently based on attributes such as gender
or race.  This is a well-known problem, but detecting \emph{where} a model behaves unfairly
is not straightforward---the input space is typically high-dimensional, and discriminatory
behaviour may only appear in certain regions.

I chose the \textit{AI Model Fairness Testing} problem from the module baseline list
because it connects search-based software engineering directly to a real-world social
concern.  The core question is simple: given a fixed evaluation budget, can a genetic
algorithm (GA) find more discriminatory inputs than a plain random search?

\section{Related Work}
Fairness in machine learning is generally examined from two angles.
Group-level metrics such as demographic parity and equalized odds~[1] measure whether a
model's predictions differ systematically across demographic groups.
Individual-level fairness, on the other hand, asks whether a single person would receive a
different outcome if only their sensitive attribute (e.g.\ gender) were changed.
Both views matter; in this project I focus primarily on the individual-level measure while
also reporting group-level metrics for a more complete picture.

From a search-based software engineering perspective, random search is the standard
baseline because it is trivial to implement and needs no parameter tuning.  The downside
is that it treats the input space uniformly and never builds on what it has already found,
which becomes inefficient when the evaluation budget is limited.
Genetic algorithms~[2], in contrast, maintain a population that gradually shifts toward
high-fitness regions through selection, crossover, and mutation.  The trade-off is that GAs
introduce extra parameters (population size, mutation rate, selection pressure) that have to
be chosen carefully---poor settings can lead to premature convergence or wasted
evaluations.

\section{Problem Definition}
Let $f(x)$ denote a classifier and $T_s(x)$ the operator that flips the sensitive
attribute of input $x$.
\begin{equation}
D(x) = |P(y{=}1 \mid x) - P(y{=}1 \mid T_s(x))|
\end{equation}
$D(x)$ is the individual discrimination score: the bigger the value, the more the
prediction depends on the sensitive attribute alone.

\section{Method}
\subsection{Genetic Algorithm}
The population at generation $t$ is
\[
\mathcal{P}^{(t)} = \{x_1, \ldots, x_N\}
\]
and the objective is to maximise $D(x)$---i.e.\ find inputs that expose the largest
fairness gap.

I use tournament selection ($k{=}3$) because it provides moderate selection pressure without
requiring the population to be globally sorted.  Crossover is uniform: each feature of the
child is copied from one of the two parents with equal probability.  Mutation adds
Gaussian noise $\mathcal{N}(0, 0.1)$ to a single randomly chosen feature with probability
equal to the mutation rate.  All generated values are clipped to the training-data feature
ranges so that the candidates stay realistic.

To keep the comparison fair, I enforce an \textbf{equal evaluation budget} for both GA and
random search.  Every call to the model (including the initial population) counts toward
the budget.  With population size $N$ and budget $B$, the GA runs for roughly
$\lfloor B/N \rfloor$ generations.

\section{Experiments}
\subsection{Setup}
\begin{itemize}
    \item Datasets: Adult, COMPAS, German Credit.
    \item Sensitive attributes: \texttt{sex} (Adult), \texttt{race} (COMPAS),
          \texttt{sex} (German).
    \item Classifier: RandomForestClassifier (scikit-learn, default parameters).
    \item Independent trials: 30 (default).
    \item Budget per trial per method: $B$ evaluations, matched between GA and
          random search.
    \item Baseline: Random Search with the same budget.
    \item Fairness metrics recorded per trial:
    \begin{itemize}
        \item Individual discrimination count ($D(x) > 0.2$),
        \item Mean individual discrimination score,
        \item Demographic parity difference,
        \item Equalized odds difference (proxy over generated samples).
    \end{itemize}
\end{itemize}

\subsection{Procedure}
For each dataset, both methods run with identical budgets and random seeds.
All four fairness metrics are recorded after every trial.
I also record convergence traces---best and mean fitness per generation (GA) or
per chunk (random search)---to see how performance develops over the budget.

Hyperparameter sensitivity is checked by varying population size
$\in \{10, 20, 40\}$ and mutation rate $\in \{0.05, 0.1, 0.2\}$
while keeping the total budget constant.

\section{Statistical Test}
I use the Wilcoxon signed-rank test~[3] at $\alpha = 0.05$ with the one-sided
alternative hypothesis GA $>$ Random Search.  The test is applied separately for each
dataset--metric pair, using the paired trial results.

\section{Reflection}
There are a few clear limitations:
\begin{itemize}
    \item The fairness metrics are computed over \emph{generated} samples, not real
          deployment data, so the estimated discrimination may not match what would happen
          in practice.
    \item Equalized odds is only a proxy here because the generated candidates have no
          ground-truth labels---I use counterfactual model predictions instead.
    \item GA performance depends on hyperparameters; although the sensitivity analysis
          covers several configurations, the search over settings is not exhaustive.
    \item Using a single classifier type (RandomForest) limits generalisability.
\end{itemize}

Possible improvements for future work include multi-objective optimisation (balancing
discrimination discovery with input realism), constraint-aware sampling to stay closer to
the real data manifold, and testing on additional classifiers.

\section{Conclusion}
This project provides a reproducible pipeline for comparing GA-based and random fairness
testing under matched budgets.  Preliminary results suggest that GA finds discriminatory
inputs more effectively, but statistical confirmation requires a full-scale run with enough
trials.  The toolchain also supports convergence analysis and hyperparameter sensitivity,
which help explain \emph{why} the GA does (or does not) outperform random search on a
given dataset.
